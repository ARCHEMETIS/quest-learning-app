# เก็บเงินจริงใน MVP: ขายอะไร เก็บยังไง โดยไม่ฆ่ายอดผู้ใช้

Type: grilling
Status: resolved (2026-07-12)
Assignee: owner+claude (session 12 ก.ค. 2026)
Blocked by: 06, 08

## Question

เจ้าของตัดสินใจแล้ว (12 ก.ค.) ว่าอยากเก็บเงินจริงภายในเดือนนี้ ไม่ใช่แค่เรื่องเล่าใน BMC และระบุทิศทางแล้วว่าเป็น **freemium — ใช้ฟรีได้ + มีชั้นจ่ายเงิน** (ไม่ใช่ donate ล้วนหรือ paywall ทั้งแอพ) — คำถามที่ต้องตอบ:

1. **เส้นแบ่งฟรี/จ่ายอยู่ตรงไหน**: ฟรีต้องพอให้แอพสนุกและยอดผู้ใช้โต (คะแนนวิชายังวัดจาก*จำนวนผู้ใช้*) แล้ว premium ขายอะไรที่คนยอมจ่ายในเดือนแรก — แชทโค้ชไม่จำกัด (สอดคล้อง limit 10 ข้อความ/วันจากผล ticket โควต้า Gemini)? หัวข้อพิเศษ? badge/ของตกแต่ง? ราคาเท่าไหร่ (ครั้งเดียว/รายเดือน)?
2. **เก็บเงินยังไงแบบงบศูนย์**: ข้อจำกัด free tier ตัด payment gateway มีค่าธรรมเนียม/รายเดือนทิ้ง — ตัวเลือกจริงคือ PromptPay QR (ฟรี, คนไทยคุ้น) + ยืนยันแบบ manual หรือกึ่ง manual ไหวไหมสำหรับคนเดียว?
3. **สมดุลกับเป้ายอดผู้ใช้**: paywall วางตรงไหนถึงไม่เพิ่ม friction ตอน onboarding (ห้ามขวางทางไปเควสแรกเด็ดขาด)
4. **เส้นเวลา**: ฟีเจอร์จ่ายเงินกินเวลา build จาก ~2 สัปดาห์ที่มี — ตัดอะไรแลก หรือปล่อยเป็น fast-follow สัปดาห์ที่ 3?

**Input จาก "ฟีเจอร์ MVP" (#06, ปิด 12 ก.ค.):** เส้นแบ่งฟรี/จ่ายเส้นแรกล็อกแล้ว — **ฟรี = active roadmap ทีละ 1 หัวข้อ, premium = หลายหัวข้อพร้อมกัน**; ตัวเลือกขายเพิ่มที่สอดคล้องของเดิม: แชทโค้ชเกิน 10 ข้อความ/วัน; ชั้นจ่ายเงินถูกจัดอยู่ Wave 2 (หลังเปิดตัว ~2 สัปดาห์แรก) ตอบข้อ 4 เรื่องเส้นเวลาไปแล้วบางส่วน

ผูกกับฟีเจอร์ MVP (#06 — ต้องรู้ก่อนว่าแอพมีอะไรถึงรู้ว่าอะไรขายได้) และโมเดลธุรกิจ (#08 — revenue story ใน BMC ควรตรงกับที่เก็บจริง จะได้เล่าใน pitch ว่า "ทดลองเก็บเงินแล้ว มีคนจ่ายจริง N คน" ซึ่งแข็งกว่าเรื่องเล่าเปล่า ๆ มาก)

## Resolution (2026-07-12)

> ยึดการตัดสินใจที่ล็อกจาก #08: **ราคา 39 บาท/เดือน จ่ายผ่าน PromptPay QR ยืนยันกึ่ง manual** (เจ้าของเคาะรายเดือน 12 ก.ค. — เดิมเดารายเทอม 99 แล้วเปลี่ยน), ฟรี = 1 หัวข้อ + แชท 10/วัน, premium = หลายหัวข้อพร้อมกัน + แชทไม่จำกัด, ชั้นจ่ายเงินอยู่ Wave 2. ด้านล่างคือแผนลงมือได้จริงของ 4 คำถาม + spec ฟิลด์ที่ส่งต่อ schema #15. **หมายเหตุรายเดือน:** รอบบิลถี่กว่ารายเทอม ~4 เท่า → verify manual บ่อยขึ้น + auto-expire runway สั้นลง (ดูคำถาม 4).

### คำถาม 1 — เส้นแบ่งฟรี/จ่ายละเอียด (ยืนยัน premium ขายอะไร)

**หลักคิด:** ฟรีต้องเป็นแอพที่ "ครบและสนุกในตัวเอง" (เพราะคะแนนวิชาวัดจากจำนวนผู้ใช้ ต้องไม่มีกำแพงกั้นคนส่วนใหญ่) — premium คือ upsell ของ "คนติดใจที่อยากได้มากกว่า" ไม่ใช่ปลดล็อกฟีเจอร์แกน

**ชั้นฟรี (ต้องพอสนุกให้ยอดโต) — ได้ครบทุกอย่างต่อไปนี้:**
- active roadmap **1 หัวข้อ** (สลับหัวข้อได้ แต่ทีละหนึ่ง — สลับแล้ว progress หัวข้อเดิมยังเก็บไว้ ไม่หาย)
- เควสรายวันเต็มรูปแบบ + XP + streak + phase progress + letter grade (แกน ml-quest ทั้งชุด)
- **แชทโค้ช AI 10 ข้อความ/วัน** (ตามลิมิตจาก #03 — เป็นเพดานฟรี ไม่ใช่ของ premium ล้วน)
- ฟีเจอร์ viral ครบ: ลิงก์ชวนเพื่อน, leaderboard, การ์ดแชร์ streak (ห้ามเอาไปขาย — พวกนี้คือเครื่องยนต์การโต ต้องฟรี 100%)
- web push กัน streak ขาด (Wave 2) — ฟรี

**ชั้น premium (39 บาท/เดือน) ปลดล็อก:**
1. **หลายหัวข้อพร้อมกัน** (active roadmap หลายอันขนานกัน) — จุดขายหลัก ตรงกับ #06/#08
2. **แชทโค้ช AI ไม่จำกัด** (ทะลุเพดาน 10/วัน) — จุดขายรอง แต่เป็น trigger ที่คนถึงบ่อยสุด
3. **Badge "ผู้สนับสนุน/Premium"** ข้างชื่อบน leaderboard + การ์ดแชร์ — ของตกแต่งต้นทุน 0 เพิ่ม perceived value และช่วย viral (คนเห็น badge บน leaderboard = social proof ว่ามีคนจ่ายจริง)

**ตั้งใจไม่ขาย (ตัดสินใจแล้ว):**
- ไม่ขายหัวข้อพิเศษ/เนื้อหา exclusive — เพราะ (ก) เนื้อหามาจากแหล่งฟรีภาษาไทย การกั้นเนื้อหาขัด VP "แหล่งฟรี" (ข) เพิ่มภาระ curate. ทุกหัวข้อเปิดให้ฟรีหมด ต่างกันแค่ "เรียนพร้อมกันได้กี่หัวข้อ"
- ไม่กั้น leaderboard/ชวนเพื่อน/แชร์ ไว้หลัง paywall เด็ดขาด (จะฆ่าการโต)
- **ราคาเดียว ไม่มี tier ย่อย** — ลด complexity ของ manual verification (คนเดียวทำไหว)

> ทำไม 3 อย่างนี้พอ: quota Gemini (#03) บอกว่าต้นทุนจริงของ premium ทั้งสองข้อ (หัวข้อเพิ่ม = pre-generate เควสเพิ่ม, แชทไม่จำกัด = call เพิ่ม) คือโควต้า AI — ดังนั้น premium ที่แท้คือ "ขายสิทธิ์ใช้ AI มากขึ้น" ซึ่ง margin ยังบวกเพราะผู้จ่ายเป็น subset เล็ก. Badge เป็นของแถมต้นทุน 0 ที่ทำให้ราคา 39 รู้สึกคุ้มขึ้น

### คำถาม 2 — เก็บเงินงบศูนย์ยังไง (flow PromptPay QR กึ่ง manual)

**หลักการ:** ไม่มี payment gateway (มีค่าธรรมเนียม/รายเดือน = ผิดข้อจำกัดงบศูนย์). ใช้ PromptPay ส่วนบุคคลของเจ้าของ (ค่าธรรมเนียม 0) + ยืนยันสลิปกึ่ง manual. คนเดียวไหวเพราะที่สเกลวิชานี้ (หลักสิบ–ร้อยคน) การตรวจสลิปคือคลิกไม่กี่ทีต่อคน

**Flow ผู้ใช้ (happy path):**
1. ผู้ใช้กด trigger จุดขาย (เช่น "เพิ่มหัวข้อที่ 2" หรือแชทครบ 10/วัน) → เปิดหน้า Premium
2. หน้า Premium แสดง: สิ่งที่ได้ + ราคา 39 บาท/เดือน + ปุ่ม "สมัคร Premium"
3. กดปุ่ม → ระบบสร้าง **payment record** สถานะ `pending` พร้อม **reference code ไม่ซ้ำ** (เช่น `LQ-<ตัวเลข/ตัวอักษรสั้น>` ผูกกับ user id)
4. แสดง **PromptPay QR** = จำนวน 39 บาท ระบุตายตัว (generate QR ฝั่ง client จาก payload มาตรฐาน EMVCo/พร้อมเพย์ ด้วยเบอร์/เลขบัตร ปชช. พร้อมเพย์ของเจ้าของ — ไม่ต้องใช้ API เสียเงิน) + โชว์ ref code ให้ผู้ใช้ใส่ในหมายเหตุการโอน (หรือให้จำ)
5. ผู้ใช้โอนผ่านแอพธนาคารตัวเอง → **อัปโหลดสลิป** (รูป) กลับเข้าแอพ → payment record อัปเดตเป็น `submitted` (แนบ URL รูปใน Supabase Storage) + timestamp
6. ผู้ใช้เห็นสถานะ "รอตรวจสอบ (ปกติภายใน 24 ชม.)" — ยังใช้ฟรีต่อได้ระหว่างรอ (ไม่ block)
7. **เจ้าของ** เปิดหน้า admin ง่าย ๆ (list payment ที่สถานะ `submitted`) → ดูรูปสลิป + จำนวน + ref → เทียบกับยอดเข้า PromptPay จริง → กด **Verify**
8. Verify → payment record เป็น `verified` + เซ็ต `premium_until = now + 1 เดือน` บน user → **flip flag `is_premium`** → ผู้ใช้ได้สิทธิทันที (แจ้งเตือน/next load เห็นผล)
9. ถ้าสลิปมั่ว/ไม่พบยอด → กด **Reject** พร้อมเหตุผล → สถานะ `rejected` → ผู้ใช้เห็นเหตุผล ลองใหม่ได้

**กลไกกัน edge case (ตัดสินใจแล้ว, ทำเท่าที่จำเป็น):**
- **สลิปปลอม:** ที่สเกลนี้เจ้าของเทียบกับยอดเข้าบัญชีจริงด้วยตา (manual) ก่อน verify — ไม่ต้องต่อ slip-verify API (เสียเงิน/ต้องขึ้นทะเบียน). ref code ช่วย match คน↔ยอด
- **จ่ายซ้ำ:** ปุ่มสมัครถ้ามี pending/submitted อยู่แล้วให้ชี้ record เดิม ไม่สร้างใหม่ซ้อน
- **หมดอายุรายเดือน:** cron/scheduled function เช็ค `premium_until < now` → flip `is_premium` กลับ false (renew = จ่ายรอบใหม่). ผู้จ่ายรายเดือนหมดอายุใน ~1 เดือน — premium เพิ่งเปิด Wave 2 (~สัปดาห์ 3) จึงยังไม่มีใครหมดอายุก่อน pitch 9 ส.ค. แต่ **runway สั้น**: ต้องทำ cron ภายใน ~1 เดือนหลังเปิด premium (ดูคำถาม 4)
- **admin auth:** หน้า admin กันด้วยการเช็ค user id = เจ้าของ (hardcode/role) — ไม่ต้องสร้างระบบ role เต็ม

**ต้นทุน = 0 จริง:** QR generate ฝั่ง client (ไลบรารีฟรี), Storage สลิปใช้ Supabase free, ยืนยันใช้แรงคน. ไม่มีค่าธรรมเนียมธุรกรรม

### คำถาม 3 — สมดุลกับยอดผู้ใช้ (paywall วางตรงไหน)

**กฎเหล็ก:** onboarding → เควสแรก **ต้องไม่เจอ paywall แม้แต่นิดเดียว**. เส้นทาง สมัคร Google → เลือกหัวข้อ/ระดับ/เวลา → เริ่มเควสแรก → ทำเควส → ได้ XP/streak = ฟรีล้วน ไม่มีการเอ่ยถึง premium เลย. ผู้ใช้ต้อง "ติด" ก่อนถึงจะเห็นข้อเสนอ

**Trigger จุดขาย (โผล่เฉพาะตอนผู้ใช้ชนเพดานฟรีเองโดยธรรมชาติ — contextual upsell ไม่ใช่ interstitial):**
1. **พยายามเพิ่มหัวข้อที่ 2** — ผู้ใช้กด "เพิ่มหัวข้อ/เริ่ม roadmap ใหม่" ทั้งที่มี active อยู่แล้ว → sheet ขึ้น: "อยากลุยหลายหัวข้อพร้อมกัน? ปลดล็อกด้วย Premium 39.-/เดือน" + ทางเลือก "หรือสลับมาหัวข้อนี้แทน (ฟรี)" — จุดนี้คือ trigger หลัก แข็งแรงสุดเพราะผู้ใช้แสดง intent ชัด
2. **แชทครบ 10/วัน** — พอผู้ใช้ส่งข้อความที่ 11 → โค้ชตอบว่า "วันนี้คุยครบโควต้าฟรีแล้ว เดี๋ยวพรุ่งนี้รีเซ็ต 🙌 หรืออยากคุยต่อไม่อั้น? Premium ปลดล็อกให้" — ยังได้ value (บอกพรุ่งนี้ได้ต่อ) ไม่ใช่กำแพงหัวร้อน
3. **(เสริม, ไม่บังคับ) หลัง milestone บวก** — เช่น streak ครบ 7 วัน หรือขึ้น leaderboard top → การ์ดฉลอง + บรรทัดเล็ก ๆ "อยากเก่งหลายด้านพร้อมกัน? ลอง Premium" — ยิงตอนอารมณ์ดี conversion สูง แต่ถ้าทำให้รก ตัดได้

**ทั้งหมดเป็น soft trigger:** ปิด/ปัดทิ้งได้ทุกจุด, ไม่มี pop-up ค้างขวางการใช้งาน, ไม่จำกัดจำนวนครั้งที่ผู้ใช้ dismiss แล้วใช้ฟรีต่อ. entry point ถาวรที่ไม่รบกวน = ปุ่ม/เมนู "Premium" ในหน้าโปรไฟล์ ให้คนที่อยากจ่ายเองหาเจอได้ตลอด

### คำถาม 4 — เส้นเวลา (scope ขั้นต่ำก่อน pitch ~9 ส.ค.)

**กรอบ:** Wave 1 (เปิดตัว ~2 สัปดาห์) = ไม่มี payment เลย เน้นโตยอดผู้ใช้. **ชั้นจ่ายเงินเป็น fast-follow สัปดาห์ที่ 3** (Wave 2). ต้องมี "คนจ่ายจริง N คน" ให้เล่าตอน pitch 9 ส.ค.

**Scope ขั้นต่ำที่เก็บเงินได้จริง (MVP payment — ทำก่อน pitch):**
- ตาราง/ฟิลด์ payment + flag premium (ดู schema #15 ด้านล่าง)
- หน้า Premium + ปุ่มสมัคร + generate PromptPay QR + ref code (client-side)
- อัปโหลดสลิป → Supabase Storage → set สถานะ `submitted`
- gating จริง 1 จุดเป็นอย่างน้อย: เช็ค `is_premium` ที่ "เพิ่มหัวข้อที่ 2" (จุดขายหลัก) + เพดานแชท
- **หน้า admin แบบดิบสุด**: list `submitted` + ปุ่ม Verify/Reject (ใช้ได้เฉพาะเจ้าของ) — นี่คือชิ้นที่ทำให้ "เก็บเงินจริง" เกิดขึ้นได้
- trigger จุดขายอย่างน้อยข้อ 1 (เพิ่มหัวข้อที่ 2) + ข้อ 2 (แชทเต็ม)

**ตัดออก/เลื่อนได้ (เพื่อให้ทัน) — ตัดสินใจแล้ว:**
- **auto-expire premium (cron flip กลับ)** — เลื่อนได้ถึงหลัง pitch (ยังไม่มีใครครบ 1 เดือนก่อน 9 ส.ค.) แต่ **เป็นการเลื่อนสั้น ๆ ไม่ใช่ยาวเหมือนรายเทอม** — ต้องทำ cron ภายใน ~1 เดือนหลังเปิด premium ระหว่างนี้ set `premium_until` เก็บไว้ก่อน
- **slip-verify อัตโนมัติ** — ไม่ทำ (manual ตลอดในเฟสนี้)
- **trigger ข้อ 3 (milestone upsell)** — nice-to-have ตัดได้
- **badge premium** — ถ้าเวลาบีบ เลื่อนไปหลัง pitch ได้ (ไม่กระทบการเก็บเงิน แต่ทำง่ายมาก แนะนำเก็บไว้เพราะช่วย social proof)
- **หลาย tier ราคา** — ไม่มีอยู่แล้ว (ราคาเดียว) = ประหยัดเวลา verify

**เหตุผลแลก:** สิ่งที่ต้อง "จริง" ก่อน pitch คือ *มีคนกดจ่าย 39 แล้วเจ้าของ verify แล้วเขาได้สิทธิ* — flow นี้ทั้งเส้นต้องเดินได้ end-to-end แม้จะดิบ. ทุกอย่างที่ไม่อยู่บนเส้นนั้น (expire, automation, tier, ตกแต่ง) เลื่อนได้หมด. เป้าเชิงตัวเลข: ขอแค่ M ≥ ~3–5 คนจ่ายจริง (เพื่อน beachhead) ก็พอเปลี่ยน pitch จาก "เรื่องเล่า" เป็น "พิสูจน์ willingness-to-pay แล้ว"

### สรุปฟิลด์สถานะการจ่าย → ส่งต่อ schema #15

**ตาราง `payments` (แนะนำ):**
- `id` (uuid, pk)
- `user_id` (uuid, fk → users) — ใครจ่าย
- `ref_code` (text, unique) — โค้ดอ้างอิงใส่หมายเหตุโอน / match คน↔ยอด (เช่น `LQ-XXXX`)
- `amount` (integer, บาท — ค่าเริ่มต้น 39) — เก็บไว้เผื่อเปลี่ยนราคา (29/59)
- `status` (text/enum) — **`pending` → `submitted` → `verified` / `rejected`**
- `slip_url` (text, nullable) — path รูปสลิปใน Supabase Storage
- `reject_reason` (text, nullable)
- `created_at`, `submitted_at`, `verified_at` (timestamptz, nullable ตามสถานะ)
- `verified_by` (uuid/text, nullable) — เจ้าของที่กด verify (audit)

**บนตาราง `users` (หรือ profile) เพิ่ม:**
- `is_premium` (boolean, default false) — flag หลักที่ทุก gating เช็ค
- `premium_until` (timestamptz, nullable) — วันหมดสิทธิ (now + 1 เดือน ต่อรอบจ่าย); ใช้ตอนทำ auto-expire
- (มาจาก #06 อยู่แล้ว: active topics — ฟรีจำกัด 1, premium หลายอัน — เช็คคู่กับ `is_premium`)

**Storage:** bucket สำหรับสลิป (เช่น `payment-slips`) — private, อ่านได้เฉพาะ owner/admin (RLS)

### จุดที่เจ้าของควรรีวิว (flag)

1. **เบอร์/เลขพร้อมเพย์ที่ฝัง QR** — ต้องเป็นบัญชีที่เจ้าของโอเคให้ผู้ใช้เห็น/รับเงินตรง. ตัดสินใจใช้ PromptPay บุคคลของเจ้าของ (ต้นทุน 0) — ยืนยันว่าใช้เบอร์หรือเลขบัตร ปชช. และสบายใจกับ privacy ระดับนี้ (เลขจะปรากฏใน payload QR ฝั่ง client)
2. **ผู้ใช้ได้สิทธิ "หลัง verify" ไม่ใช่ทันทีที่อัปสลิป** — เลือกแบบนี้เพื่อกันสลิปปลอม แต่แลกด้วย delay (≤24 ชม.). ถ้าอยากลด friction ช่วงโปรโมต อาจให้สิทธิทันทีที่ `submitted` แล้วค่อย revoke ถ้าสลิปมั่ว — **ยังไม่เลือกแบบนี้** (เสี่ยงโดนโกงสิทธิ AI ฟรี) แต่ flag ให้พิจารณา
3. **ราคา 39 บาท/เดือน (เจ้าของเคาะ 12 ก.ค.)** — ตาม #08; ปรับเป็น 29 (friction ต่ำ) หรือ 59 (เท่า Spotify student) ได้ทันทีเพราะ `amount` เก็บเป็นฟิลด์ ไม่กระทบ schema
4. **Auto-expire runway สั้น (สำคัญขึ้นเพราะรายเดือน)** — ผู้จ่ายหมดอายุใน ~1 เดือน (รายเทอมคือ ~4 เดือน) แม้ยังไม่มีใครหมดก่อน pitch 9 ส.ค. แต่ต้องทำ cron flip กลับ**ภายใน ~1 เดือนหลังเปิด premium** ไม่งั้น premium กลายเป็นฟรีตลอดชีพ — เลื่อนได้แค่สั้น ๆ ไม่ใช่ยาวเหมือนตอนรายเทอม
5. **badge premium อาจถูกตัด** ถ้าเวลาบีบ — flag ว่ามันช่วย social proof/viral มาก แนะนำเก็บ ถ้าตัดคือตัดชั่วคราว
6. **ต้นทุน AI ของผู้จ่าย** — "แชทไม่จำกัด + หลายหัวข้อ" = ผู้จ่ายกินโควต้า Gemini มากกว่าคนฟรีหลายเท่า. ที่สเกลวิชานี้ไม่เป็นไร (ผู้จ่ายน้อย) แต่ถ้า premium โตเร็ว ต้องเฝ้า RPM/RPD (#03) — อาจต้องตั้ง soft cap แชท premium (เช่น 100/วัน กัน abuse) ไว้ในใจ
