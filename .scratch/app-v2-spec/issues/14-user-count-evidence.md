# วิธีนับและโชว์ "ผู้ใช้จริง" เป็นหลักฐานให้อาจารย์

Type: grilling
Status: resolved (2026-07-12)

## Question

ล็อกอินถูกล็อกเป็น Google-only แล้ว (ผล [วิธีล็อกอินและ onboarding flow](04-login-onboarding.md)) ทุกบัญชีคือคนจริงที่นับได้ — คำถามที่เหลือ: (1) นิยาม "ผู้ใช้จริง" ที่รายงานอาจารย์ — นับทุกบัญชีที่สมัคร หรือเฉพาะคนที่ทำเควสอย่างน้อย 1 อัน? (2) โชว์ยังไงให้น่าเชื่อ — หน้า /stats สาธารณะในแอพ (ยอดผู้ใช้ + กราฟโต), screenshot Supabase dashboard, หรือทั้งคู่? (3) metric รองที่ทำให้ pitch แข็งขึ้น (DAU, จำนวนเควสที่ทำเสร็จ, streak เฉลี่ย, จำนวนคนจ่ายเงิน) เก็บอะไรบ้างตั้งแต่วันแรกถึงจะมีข้อมูลย้อนหลังตอน pitch

## Resolution (2026-07-12)

ตัดสินใจแทนเจ้าของตามบริบทที่ล็อกไว้ (Google-only login, MVP metric = XP/streak/quest, pitch ~9 ส.ค. 2026). จุดที่ควรรีวิวรวมไว้ท้าย section.

### 1. นิยาม "ผู้ใช้จริง" — รายงาน 2 ระดับ ยึด activated เป็นตัวหลัก

รายงานอาจารย์ **2 ระดับคู่กันเสมอ** เพื่อความโปร่งใส (อย่ารายงานตัวเดียวลอย ๆ):

- **(ก) Registered** = ทุกบัญชี Google ที่สมัคร = ทุกแถวใน `auth.users` / โปรไฟล์. เพราะ login เป็น Google-only ไม่มี guest/anonymous ตัวเลขนี้ "สะอาด" อยู่แล้ว (ไม่มีบัญชีผี, ไม่มี email ปลอม — Google ยืนยันตัวตนมาให้)
- **(ข) Activated** = บัญชีที่ทำเควสสำเร็จ **อย่างน้อย 1 อัน** (มีแถวใน quest-completion / phase progress > 0). คือคนที่ "ลองใช้จริง" ไม่ใช่แค่กดล็อกอินแล้วปิด

**ตัวหลักที่ชูในรายงาน = Activated** เหตุผล:
- โจทย์วิชาธุรกิจวัด "ผู้ใช้จริง" ไม่ใช่ "ยอดสมัคร" — activated สื่อถึง product-market fit และ engagement ซึ่งเป็นสิ่งที่อาจารย์/กรรมการ pitch อยากเห็นจริง ๆ
- ป้องกันข้อครหา "ปั๊มยอดจากเพื่อนที่กดสมัครให้แล้วไม่ใช้" — โดยเฉพาะเมื่อการโตหลักมาจากลิงก์ชวนเพื่อน (mechanic ที่อาจดันยอด registered สูงกว่าคนใช้จริง)
- Activation rate (activated ÷ registered) เป็น metric ที่ดูเป็นมืออาชีพในสไลด์ pitch เอง

รายงาน registered ควบไว้เป็นตัวเลข headline ที่ใหญ่กว่า ("สมัครแล้ว N คน, ใช้งานจริง M คน") — ได้ทั้งเลขสวยและความน่าเชื่อ

> เกณฑ์ activated ที่เลือก = "ทำเควสเสร็จ ≥ 1" (ไม่ใช่แค่ผ่าน onboarding) เพราะ onboarding จบใน <1 นาทีและ starter quest ขึ้นทันที เกือบทุกคนที่ล็อกอินจะผ่าน onboarding อยู่แล้ว จุดที่แยกคน "ลองจริง" ออกคือทำเควสจบอย่างน้อย 1 อัน

### 2. โชว์ยังไง — เอาทั้งคู่ (หน้า /stats สาธารณะ + screenshot Supabase)

**ตัดสินใจ: เอาทั้งสองอย่าง** — บทบาทต่างกัน ไม่ซ้ำซ้อน:

- **หน้า /stats สาธารณะในแอพ** = ของโชว์หลัก เข้าถึงได้ทุกคน (อาจารย์เปิดลิงก์ดูเองได้ตลอด), ดูมืออาชีพ, และเป็น social proof ให้ผู้ใช้ใหม่ที่แชร์กันปากต่อปาก ("มีคนใช้ตั้ง N คนแล้ว") — เสริม mechanic การโตโดยตรง
- **Screenshot Supabase dashboard** = หลักฐานยืนยันว่าเลขในหน้า /stats ไม่ได้ปั้น (raw row count จาก DB จริง) แนบในสไลด์/รายงานตอน pitch เท่านั้น ไม่ต้องโชว์ในแอพ

**→ สรุปให้ ticket #10 (design brief): ต้องมีหน้า /stats — ต้องแจ้งเพื่อนที่ออกแบบ UI ให้เพิ่มหน้านี้เข้า scope** (ปกติ ticket นี้ห้ามแก้ #10 — แค่ flag ให้เจ้าของไปแจ้งเอง). หน้า /stats แสดง:

1. **ตัวเลข headline 2 ตัว**: "สมัครแล้ว N คน" (registered) + "ใช้งานจริง M คน" (activated) — activated เด่นกว่า
2. **กราฟการเติบโตสะสม** (cumulative) ของยอดผู้ใช้ตามวัน — เส้นโตขึ้นเรื่อย ๆ คือภาพที่ pitch ต้องการ; ต้องมีข้อมูลย้อนหลังตั้งแต่วันเปิดตัว (ดูข้อ 3)
3. **ตัวเลขรองแบบ social-proof**: จำนวนเควสที่ทำเสร็จรวมทั้งแอพ, streak สูงสุด/เฉลี่ย (เลือกโชว์เฉพาะที่ดูดี)
4. อัปเดตใกล้ real-time (query ตรงจาก Supabase ตอนโหลดหน้า หรือ cache รายชั่วโมงพอ) — **ไม่ใส่ข้อมูลส่วนตัว/ชื่อผู้ใช้** ในหน้าสาธารณะ (ยกเว้น leaderboard ที่เป็นคนละหน้าและผู้ใช้ยินยอมโดยปริยายจากการเล่น)

โน้ตเทคนิค: /stats เป็น read-only aggregate query ต้นทุน 0 (อยู่ใน Supabase free tier) ไม่กินโควต้า Gemini เลย

### 3. Metric รองที่ต้องเริ่มเก็บ "ตั้งแต่วันเปิดตัว"

**หลักการสำคัญ:** metric พวกนี้เป็น time-series — ถ้าไม่บันทึก event ตั้งแต่วันแรก จะ **ไม่มีทางย้อนสร้างกราฟย้อนหลัง** ตอน pitch (~9 ส.ค.) ได้เลย ต้องเก็บ timestamp ทุก event ตั้งแต่ deploy วันแรก แม้ยังไม่มีหน้าแสดงผลก็ต้องเก็บ raw ไว้ก่อน. Schema จริงเป็นงานของ ticket #15 — ตรงนี้ระบุคร่าว ๆ ว่าเก็บที่ไหน:

| Metric | เก็บที่ไหน (คร่าว, ยืนยันใน #15) | หมายเหตุ |
|---|---|---|
| **Registered total** | นับแถว `profiles` / `auth.users` (มี `created_at` ต่อคน) | ได้ฟรีจาก Auth อยู่แล้ว |
| **Activated total** | มีอย่างน้อย 1 แถวใน `quest_completions` | นิยามข้อ 1(ข) |
| **DAU** | ตาราง event log เช่น `activity_log(user_id, event_type, created_at)` หรือ field `last_active_at` ใน `profiles` (อัปเดตทุกครั้งที่เปิดแอพ) | DAU = distinct user_id ที่ active ในวันนั้น; ต้องมี timestamp ต่อ session/วัน ไม่งั้นสร้างกราฟ DAU ย้อนหลังไม่ได้ — **นี่คือตัวที่พลาดง่ายสุด** |
| **จำนวนเควสเสร็จ** | `quest_completions(user_id, quest_id, completed_at)` | มี `completed_at` เพื่อทำกราฟรายวัน + นับ total |
| **Streak เฉลี่ย** | field `current_streak` / `longest_streak` ใน `profiles` (มีจากของเดิม ml-quest) | เฉลี่ยคำนวณตอน query; ถ้าอยากได้ streak ย้อนหลังต้อง snapshot รายวัน |
| **จำนวนคนจ่ายเงิน** | field `is_premium` / `plan` + `subscribed_at` ใน `profiles` หรือตาราง `subscriptions` | Wave 2 แต่ **ต้องมี `subscribed_at` timestamp ตั้งแต่คนแรกที่จ่าย** ไม่งั้น conversion timeline หายหมด |

**คำเตือนที่ต้องเน้นกับเจ้าของ:** อย่างน้อยต้องมี **timestamp ต่อ event** (`created_at`, `completed_at`, `last_active_at`, `subscribed_at`) ตั้งแต่ commit แรกที่ deploy — ถ้า schema #15 ลืมใส่ timestamp พวกนี้ ตอน pitch จะโชว์ได้แค่ "ยอด ณ วันนี้" แต่โชว์ "กราฟโต" ไม่ได้ ซึ่งกราฟโตคือหัวใจของ pitch วัดผู้ใช้จริง. ทางถูกที่สุดคือทำ `activity_log` เบา ๆ 1 ตาราง (user_id, event_type, created_at) เก็บทุก event สำคัญไว้ก่อน แล้วค่อย aggregate ทีหลัง

### จุดที่ควรรีวิว (flag ให้เจ้าของ)

1. **เกณฑ์ activated = "ทำเควสเสร็จ ≥ 1"** — ถ้าอาจารย์นิยาม "ผู้ใช้จริง" เข้มกว่านี้ (เช่น active ≥ 2 วัน) ต้องปรับ; ควรเช็กกับอาจารย์ว่ายอมรับนิยามนี้ไหมก่อน pitch
2. **หน้า /stats กระทบ scope ของ ticket #10** — ต้องแจ้งเพื่อนที่ออกแบบ UI ให้เพิ่มหน้านี้ (ticket นี้ไม่ได้แก้ #10 ให้; เจ้าของต้องไปแจ้ง/อัปเดต #10 เอง)
3. **`activity_log` เป็นตารางที่ยังไม่มีในสเปก** — ต้องยืนยันใน ticket #15 (schema) ว่าจะเก็บ event log แบบไหน; ถ้าตัดออกเพื่อความง่าย ต้องยอมรับว่ากราฟ DAU ย้อนหลังจะทำไม่ได้
4. **Privacy หน้า /stats** — โชว์เฉพาะ aggregate ห้ามหลุดข้อมูลส่วนตัว; leaderboard (แสดงชื่อ/รูป) เป็นคนละหน้าและควรมีตัวเลือก opt-out — เป็นประเด็นที่กรรมการวิชาธุรกิจอาจถาม
5. **จำนวนคนจ่ายเงินเป็น Wave 2** — ตอน pitch ~9 ส.ค. อาจมีผู้จ่ายน้อยหรือยังไม่มี; ต้องเตรียมเล่า conversion เป็น "เป้า/แนวโน้ม" แทนตัวเลขจริงถ้ายังไม่เปิดชั้นจ่ายทัน
